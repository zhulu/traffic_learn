# 📝 实验记录：加密流量识别基准方案 (Baseline)

## 1. 实验目标与背景
[cite_start]本阶段任务旨在验证 **Scenario A** 的第一阶段 [cite: 131, 134][cite_start]：即在复杂的加密流量中，仅利用**流的时间相关特征**区分常规加密流量（Non-VPN）与隧道加密流量（VPN） [cite: 8, 39, 135]。

## 2. 特征抽取方案 (Feature Engineering)
我们直接从原始 PCAP 报文序列中提取特征，利用报文的时间和长度信息构建时序张量。

* [cite_start]**流定义与汇聚**：基于五元组（源IP、目的IP、源端口、目的端口、协议）识别双向流 [cite: 140, 145]。
* [cite_start]**截断策略**：严格遵循论文建议的 **15s 流量超时**，仅提取每个双向流的前 **32 个报文** [cite: 207, 209]。
* [cite_start]**特征维度**：每个报文提取 3 个核心维度，构成 $32 \times 3$ 的特征矩阵 [cite: 87, 89]：
    * **报文长度 (Length)**：反映应用层数据载荷大小。
    * [cite_start]**到达间隔时间 (IAT)**：反映流量的发送节奏与心跳逻辑 [cite: 87, 89]。
    * [cite_start]**方向 (Direction)**：1 表示前向 (Forward)，0 表示后向 (Backward) [cite: 145]。
* **数据预处理**：
    * **长度归一化**：$Length / 1500$ (基于标准 MTU)。
    * **时间对数化**：$\log(1 + IAT)$ (缓解 IAT 分布的长尾效应，提升模型稳定性)。

## 3. 模型网络架构 (Model Architecture)
采用 **1D-CNN + Transformer** 的双分支融合网络。

| 分支名称 | 核心算子 | 逻辑功能 |
| :--- | :--- | :--- |
| **1D-CNN 分支** | 2层 Conv1d (Kernel Size=3) | 捕捉报文序列间的局部空间相关性（如特定协议的握手长度序列）。 |
| **Transformer 分支** | 2层 EncoderLayer (4-Head) | 通过自注意力机制捕捉 15s 全局时间节奏 (IAT) 的依赖关系。 |
| **特征融合层** | Concatenation + MLP | 拼接局部与全局特征，通过 Sigmoid 输出 VPN 概率。 |

## 4. 训练方案 (Training Scheme)
* [cite_start]**任务类型**：二分类 (VPN vs. Non-VPN) [cite: 135]。
* **数据集划分**：
    * **比例**：80% 训练集，20% 验证集。
    * **策略**：分层随机抽样 (Stratified Split)，保持各类别样本比例均衡。
* **超参数设置**：
    * **损失函数**：BCEWithLogitsLoss。
    * **优化器**：Adam (Learning Rate = 0.001)。
    * **Batch Size**：1024 (适配 NVIDIA RTX 3090)。
    * **训练轮数**：30 Epochs。

## 5. 基准结果分析 (Baseline Results)
[cite_start]以下为验证集上的初始实验结果 [cite: 585-587]：

| 指标 (Metric) | 数值 (Value) |
| :--- | :--- |
| **准确率 (Accuracy)** | **0.8633** |
| **VPN 召回率 (Recall)** | **0.5821** ⚠️ |
| **宏平均 F1 (Macro F1)** | **0.7864** |
| **最终验证集 Loss** | **0.3536** |

### 🔍 结论与瓶颈分析
1. **类别不平衡影响**：Non-VPN 样本 (1375) 与 VPN 样本 (402) 的比例约为 3.4:1，模型存在明显的预测偏向性。
2. **识别率瓶颈**：VPN 的低召回率 (0.58) 表明当前的基准方案在没有加权处理的情况下，难以有效捕捉被隧道封装掩盖的特征细节。
3. **改进方向**：需引入有符号长度特征增强、损失函数加权 (Cost-Sensitive) 以及更强的数据增强手段。
