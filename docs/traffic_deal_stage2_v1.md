# 📝 实验记录：加密流量识别应用细分 (Stage 2 - Non-VPN Expert)

## 1. 实验目标
在第一阶段（VPN vs. Non-VPN）成功分离出常规流量的基础上，针对 **Non-VPN** 流量进行更深层次的任务细分，将其精确归类为 7 种特定的互联网应用。

## 2. 任务定义与标签映射
* **任务类型**：多分类 (7-Class Classification)。
* **应用类别 (STAGE2_LABELS)**：
    1. Web Browsing (网页浏览)
    2. Email (邮件)
    3. Chat (聊天)
    4. Streaming (流媒体)
    5. File Transfer (文件传输)
    6. VoIP (网络电话)
    7. P2P (点对点下载)
* **数据集筛选**：仅从 `samples.npz` 中提取 `label1 == 'NonVPN'` 的流样本进行训练。

## 3. 特征工程：沿用双轨融合架构
为了保持特征提取的一致性，继续采用 Stage 1 的高效特征体系：
* **微观序列 (Sequence)**：前 32 个包的“有符号长度”序列与对数化 IAT 序列。
* **宏观统计 (Global Stats)**：流生命周期（15s 内）的 10 维统计特征（均值、方差、峰值、吞吐量）。
* **归一化处理**：所有统计特征在输入 MLP 前均经过 $\log(1+x)$ 处理，确保梯度稳定。

## 4. 模型架构优化 (Model Refinement)
* **输出层调整**：模型 `VPNClassifier` 的最后一层全连接维度由 $1$ 修改为 $7$。
* **损失函数切换**：由二分类的 `BCEWithLogitsLoss` 切换为适用于多分类的 `CrossEntropyLoss`。
* **预测逻辑**：通过 `torch.max(out, 1)` 获取 7 个 Logits 中概率最大的索引作为预测类别。



## 5. 训练策略与技术瓶颈突破
* **数据分层采样 (Stratified Split)**：通过 `sklearn.model_selection` 确保 7 个类别在训练集和验证集中的比例分布一致。
* **空类别鲁棒性处理**：
    * **问题**：部分类别（如 P2P）样本极度稀缺，验证集中可能出现“零样本”情况，导致 `classification_report` 维度崩溃。
    * **对策**：在 `utils.py` 中显式指定 `labels=[0, 1, 2, 3, 4, 5, 6]`，强制锁定报告维度，确保 7x7 混淆矩阵始终完整生成。

## 6. 实验结果初步复盘 (Current Status)
* **收敛情况**：50 轮训练后，验证集准确率 (Val Acc) 达到约 **0.69 - 0.71**。
* **性能评估**：
    * **对角线优势**：`VoIP` 和 `Streaming` 等强统计特征类别表现稳健。
    * **混淆高发区**：`Web Browsing` 与 `Chat` 存在特征重叠，由于两者均包含大量的短交互报文。
    * **数据长尾效应**：`P2P` 等类别由于样本量不足，对宏平均 F1 (Macro F1) 拉低明显。



### 💡 后续优化思路
1. **数据增强**：针对样本量少于 100 的类别（如 P2P），考虑在特征层引入过采样或噪声增强。
2. **模型调优**：目前的 0.70 精度距离基准仍有提升空间，下一步可考虑增加 Transformer 的层数或调整 Dropout 比例，以缓解模型在多分类下的欠拟合。
